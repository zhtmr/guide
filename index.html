===== git - git init = 깃으로 관리하겠다. - git status = 상태보기( 빨간색 : 변경 사항은 표시되지만 아직 추적은 하지 않음) - git add . = 의미있는 작업을 했다고 판단했으면 사진을 찍기 전에 stage로 올리기 (초록색) - git commit -m "" = 사진찍어서 추적 준비완료 시키기. - git checkout HEAD~ = 커서를 예전 사진으로 옮기기. <=> git checkout master [checkout 하기전에 add, commit 하고 하기] - git merge lab1 = 합칠 곳으로 checkout 한다음 합칠 branch적으면됨. - git reset 옵션 HEAD~ (~갯수만큼 뒤로) = branch pointer 이동(commit 취소) (option : hard(수정 내용 전부삭제), soft(스테이지에 그대로 있기 : 빠뜨린 내용 추가하려고 할때, commit 취소 ), mixed(스테이지에서 내려오기 : add 취소)) - git stash -u = 작업하던 내용을 잠깐 내 호주머니같은곳에 보관하기 - git stash pop = 가장 최근 stash된것 가져오기(list에서 지우면서 가져오기) - git stash apply [stash id] = stash된것 가져오기(list에서 안지우고 가져오기) - git stash save -u = untracked file 도 주머니에 넣기 - git stash list = stash 목록 - git stash drop [stash id] = stash 목록 지우기 - git checkout . = unstaged 수정내용 모두 삭제하기 - git clean -fd = untracked file 과 directory 를 모두 삭제하기 ===== git hub - git remote add origin http://xxx/ = git hub 서버에 remote 저장소 등록하고 이름은 origin으로 만들기 - git push -u origin master = local에서 작업한 내용을 git hub origin에 추가하기 [ origin/master ]가 만들어짐. git hub의 상태를 추적하는 branch pointer. 다른사람이 push하거나 다른곳에서 push하게되면 갈라지게됨.(remote에서 변경사항 fetch한 후에 master branch와 merge해야됨.) - git clone https://xxx = 외부에서 작업한 파일을 집에서 클론 할 폴더에다가 클론하기(최초한번만). 1. 이후는 변경사항만 받기. (git fetch origin master). 2. clone한 파일과 내 로컬 파일 merge하기 (git merge origin/master) * git pull origin master : fetch와 merge 한번에 할 수 있음. 충돌날 경우 수정 후 merge * Git의 '커밋' 작업은 '작업 트리'에 있는 변경 내용을 저장소에 바로 기록하는 것이 아니라 그 사이 공간인 '인덱스'에 파일 상태를 기록(stage - 스테이징 한다고 표현하기도 합니다)하게 되어 있습니다. 따라서 저장소에 변경 사항을 기록하기 위해서는, 기록하고자 하는 모든 변경 사항들이 '인덱스'에 존재해야 합니다. 예를 들어, 10개의 파일을 수정했지만 그 중에 7개만 저장소에 공개하고 싶을 때를 생각해 보세요. 변경한 10개의 파일 중 7개를 선택하는 작업이 바로 '인덱스에 등록' 또는 '스테이징(stage)'이라 표현하는 작업 입니다. 이렇게 인덱스란 공간(가상이지만요!)이 중간에 있는 덕분에 작업 트리 안에 있는 커밋이 필요 없는 파일들을 커밋에 포함하지 않을 수 있고, 파일에서 내가 원하는 일부 변경 사항만 인덱스에 등록해 커밋할 수 있습니다. * git-gui에서 그래프 확인가능. * commit을 하면 branch pointer가 바뀌고 checkout하면 HEAD가 바뀜. * 작업 후에 status를 쳐서 변경사항이 있는지, add,commit을 해야되는지 확인. * merge 할때 HEAD를 master로 checkout한다음 merge한다. (변경사항이 겹치지않으면 자동병합됨.) (변경사항이 겹치면 merge되기 전에 수정할 부분을 표시해 준다. 표시된 부분을 수정 후에 add, commit하면 된다.) * lab1에서 작업하다가 master에서 작업해야 될 경우, lab1의 내용을 add한 후에 stash로 주머니에 가지고 있다가 master로 checkout 해서 작업 완료 후에 commit 후 lab으로 다시 돌아와서 stash pop으로 붙여넣기. * remote 저장소에 oigin/master pointer가 있는 이유 : 집에서 github 파일을 클론으로 받을 때 완충작용(덮어쓰기 방지). remote 저장소의 현재 commit 상태를 나타내줌. * 시나리오 학원에서 작업 후 add, commit 해서 github로 push ==> 집에서 github 파일 clone 후 fetch+merge(pull)한다. * 따로 테스트하고 싶을때는 lab branch따로 만들어서 해야 pull해서 올때 충돌나지않음. master branch는 내려받는 용도로 사용. 테스트하다가 중간에 master로 갈때는 작업내용을 stash하고 가야 충돌나지않음. * https://docs.google.com/document/d/1cnJGSGYFCCgfwz5C2M33Q_OUV_6ubCvqerqLfLSBNqA/edit 수업 내용 add, commit 해서 git hub에 올림 [dropbox 대체] working directory | stage | local repository 준비 ====> add ====> commit reset : [ hard ] [ mixed ] [ soft ]
